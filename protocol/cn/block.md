# Septopus Block (地块) 协议

**Septopus 引擎** 将“地块 (Block)”定义为世界中最基本、不可分割的**原子空间容器**。它本质上是一个地理位置槽位，用于收纳各种不同来源的内容。

地块与特定组织协议（如 SPP）并无必然联系。它既可以承载通过标准方式部署的附属物，也可以承载通过特殊的 **`spp` 附属物**（作为解析代理）部署的复杂弦粒子结构。

## 1. 核心属性 (Core Properties)

地块不仅仅是一个视觉单元；它代表着数字资产/虚拟地产，并承载着状态和所有权元数据。

- **唯一所有权 (Unique Ownership)**：一个地块在任何特定时间只能有一个所有者。但是，一个玩家（所有者）可以拥有多个地块。
- **经济资产 (Economic Asset)**：地块是生态系统内的可交易实体。
- **遗弃与重新认领 (Abandonment & Re-claiming)**：
    - 一个地块可以由其所有者自愿舍弃。一旦被舍弃，它就会进入公共池，任何玩家都可以去认领它。
    - **衰变系统 (Decay System)**：如果一个地块在长达 100 个“游戏世界年”内没有收到任何更新或交互指示，它将被分类为“荒废 (Abandoned)”状态，并自动变为任何玩家均可认领的状态。
- **地理属性 (Geography)**：地块可以定义自己的基础海拔高度，这使得当它与相邻地块拼接连结时，能够创造出多样化的地形（如山脉、山谷）。
- **状态配置 (State Configuration)**：地块可以存储编码的状态标记，以实现预定义的宏观功能（例如：将某地块指定为“安全区 (Safe Zone)”或“活动区 (Event Area)”）。

## 2. 与附属物的关系 (Relationship with Adjuncts)

虽然地块定义了稳固的地面和世界的绝对坐标，但**附属物 (Adjuncts)** 提供了交互式的对象和动态内容。

- 所有的附属物在数学上都是相对于其父级“地块”的坐标系统进行定位的。
- 地块充当了空间锚点。如果一个地块被移动，所有锚定在它上面的附属物都会随之一起移动。

## 3. 地块 raw 数据格式(规范 / Normative)

一个地块的内容是一个**五元数组**:

```
[ elevation, status, adjuncts, animations, game ]
```

| 槽 | 字段 | 类型/缺省 | 说明 |
|---|---|---|---|
| 0 | `elevation` | number,缺省 `0` | 基础海拔(米)。地面与全部附属物整体抬升——与相邻地块的坡道(b4 slope)拼接即可造山谷/台地 |
| 1 | `status` | number,缺省 `1` | 状态位(保留;`1` = 正常) |
| 2 | `adjuncts` | `[[typeId, [行…]], …]` | 附属物分组:按类型 id 分组的 raw 行列表。**逐类型槽位规范见 [adjunct-types.md](adjunct-types.md)** |
| 3 | `animations` | 数组,缺省 `[]` | 地块级动画(保留) |
| 4 | `game` | number,缺省 `0` | **可玩标记**:`≥1` = 本地块允许进入 Game 模式(区域门控;富游戏声明在 b8 game trigger 上,此位仅粗粒度枚举用) |

- 空地块 = `[0, 1, [], [], 0]`;实现必须容忍缺槽并取缺省值。
- **序列化不变量**:运行时派生实体(SPP/motif 展开物、spawn 生成物)**永不**写回
  raw——只有源行持久化。行数预算(实现建议 64/块)只计 authored 行。
- 本地草稿(编辑未发布)覆盖同坐标的规范数据;发布后内容可进入内容寻址存储(CID)。

## 4. 存储与扩展 (Storage and Scaling)

每个地块都可以根据需要序列化为特定的数据格式（如 SPP 字符串）。正是由于地块的“原子”性质，引擎可以在单地块粒度的基础上动态加载或卸载（分片 Sharding）世界内容，从而允许无限扩展的环境，而不会使得客户端内存溢出崩溃。
