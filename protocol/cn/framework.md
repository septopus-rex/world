# Septopus 引擎架构框架协议 (Architecture Framework Protocol)

**Septopus 引擎** 是一个以 **附属物 (Adjuncts)** 为物理原子构建的 3D 内容执行环境。引擎核心处理附属物的生命周期、渲染与交互，而对内容的抽象组织则由上层协议完成。

**String Particle Protocol (SPP)** 是一套语义空间组织协议。它通过“弦”的描述将复杂的 Adjuncts 组合进行坍缩，从而实现 AI 原生的 3D 生成与分发。在 Septopus 引擎中，SPP 逻辑被封装在一个特殊的 **`spp` 元附属物 (Meta-Adjunct)** 内部，通过它来解析并还原复杂的空间结构。

本框架协议概述了 Septopus 引擎的核心架构原则、坐标系统以及 SPP 协议在数据管线中的应用。

## 1. 全局坐标系统 (Global Coordinate Systems)

SPP 严格定义了多个相互连接的坐标系统，以管理数据效率和渲染逻辑。SPP 中一个关键的区别是 **Z轴代表垂直（上/下）方向**（即玩家站立的方向），这与默认的 Three.js 坐标系不同。

| 系统 | SPP 符号 | 主要功能 |
|---|---|---|
| **地块坐标 (Block Coordinates)** | `A` 系统 | 相对于单个地块原点的局部坐标。用于在单块土地上对附属物进行高压缩比、可重复的存储。 |
| **世界坐标 (World Coordinates)** | `B` 系统 | 全局绝对坐标。用于将多个地块拼接在一起，并确定动态加载（视锥体剔除）的区域。 |
| **屏幕坐标 (Screen Coordinates)** | `C` 系统 | 2D 视口投影。用于处理用户输入（鼠标点击、触摸事件）以及渲染 2D 小地图。 |

## 2. 运行模式 (Operating Modes)

引擎必须能够根据需要在不同的预定义环境状态之间可靠地切换。SPP 协议在该模式切换中提供了数据组织边界。

### 2.1 模式定义表

| 模式 | 状态描述 | 交互能力与限制 |
|---|---|---|
| **正常浏览 (Normal/Browse)** | 注册化身的默认世界探索模式。 | 完全的移动能力，可与激活的触发器和对象进行交互。 |
| **编辑 (Edit)** | **独占式**实时世界构建。 | 启用下述 **“独占式会话”** 规则，允许修改单个地块数据。 |
| **游戏 (Game)** | 锁定的叙事或交互式模拟。 | 将渲染严格集中在相关的地块上；限制玩家离开指定的模拟边界。 |

### 2.2 编辑模式核心准则 (Exclusive Session Rules)

为了确保数据安全性与协议一致性，编辑模式必须遵循以下强制性准则：

1.  **会话锁定 (Session Locking)**：进入编辑模式时，引擎必须根据玩家位置探测并锁定唯一的 `activeBlockId`。在会话结束前，该 ID 不得随玩家移动而自动变动。
2.  **交互物理隔离 (Interaction Isolation)**：引擎必须对射线检测（Raycast）执行归属权校验。只有属于当前激活地块的附属物（Adjuncts）才允许被选中、移动或修改。对非目标块的点击应视为无效或执行“取消当前选中”。
3.  **坐标对齐协议 (Alignment Protocol)**：编辑过程中涉及的偏移量（Offset）计算必须严格执行 1-based 转换算法，确保所有视觉辅助工具（如高亮围栏）与地块地表瓦片物理对齐。

## 3. 数据流与表示状态 (Data Flow & Representation States)

为了将高维度内容组织与低维度执行解耦，系统将数据分为 **“拟存在 (Pseudo-existence)”** 与 **“物理存在”** 两个阶段。

### 3.1 核心数据定义

| 数据标识 | 含义 | 物理形态 |
|---|---|---|
| **SPP_IPFS_A** | **弦粒子全状态集 (Genome)** | 存储在 IPFS 上的全局全量协议定义，包含弦粒子的所有 Face Options 与连接规则。它是“全状态”的语言系统。 |
| **SPP_DATA** | **空间矩阵坍缩态 (Blueprint)** | 存储在 `spp` 附属物内部的具体空间矩阵数据。记录了某局部区域内弦粒子的确定性坍缩状态。 |

### 3.2 数据流转管线 (The Announcer Model)

1.  **坍缩与存储 (Collapse)**：AI 或开发者利用 SPP 协议将 3D 内容坍缩，以 `SPP_DATA` 形式持久化。
2.  **定位与挂载**：在 Block 的指定空间槽位部署一个 **`spp` 附属物**，并向其注入上述 `SPP_DATA`。
3.  **解析与实例化 (Unfold)**：`spp` 附属物作为“播音员”，套用完整的 **`SPP_IPFS_A`** 规则，按照 **`SPP_DATA`** 的坍缩方式，并在本地解析出应存在的 **其他附属物** 列表。
4.  **展示 (Display)**：解析出的附属物列表被交付给引擎渲染管线。最终用户看到的 3D 显示是由这些派生出的附属物共同构成的。

## 4. 区块链环境同步 (Blockchain Environmental Sync)

Septopus 引擎驱动的环境可以将其大气和时间条件直接动态绑定到宿主区块链的心跳上。

- **时间系统 (Time System)**：世界内的时间（影响日夜循环或老化机制）是由**区块链的当前区块高度**确定性地计算得出的。
- **天气系统 (Weather System)**：世界内的天气（雨、雾、风暴）是使用**区块链的区块哈希值**作为随机化种子按程序生成的。

## 5. 引擎架构与帧同步 (Engine Architecture & Frame Sync)

为了保证流畅的 3D 性能（维持 60FPS），Septopus 引擎采用严格的**帧同步（Tick）**架构。

- **异步解耦 (Asynchronous Decoupling)**：诸如 IPFS 网络获取或解析大型 3D 模型 `.glb` 文件等繁重任务会被推入异步队列。
- **微任务处理 (Micro-tasking)**：主渲染循环（`tick`）每帧只消耗队列中有限数量的切片任务。这确保了主 3D 渲染线程永远不会被阻塞超过 16 毫秒，从而防止在处理庞大数据加载时 UI 出现冻结卡顿。
