# 物理碰撞与检测系统 (Physics & Collider)

Septopus World 为了追求海量实体下的极简运行性能，没有直接引入如 Cannon.js 这类全套真实物理引擎。它的刚体碰撞交互采用了一套自研的基于射线检测的 **“高度差探针与 Stop 物件逻辑”**。

## 1. 核心碰撞检测逻辑：阻拦器 (Stop) 策略

在通常的环境中，所有模型网格都是可穿透的幻影。只有当这个附属物在数据位里被标记为一个**阻拦器 (Stop Object)** 时，它才会具有物理碰撞体积。

### 判断：水平拦截机制
水平方向的判断由控制输入产生。玩家每向前走动一步，探测器将向步进方向发射极短测距的射线 (Raycaster)。
*如果碰撞检测到的 `hit` 面来自于有 `stop` 位标记的对象并且它的接触点高于玩家的行走跨越度 (Span)，运动指令就会被物理引擎立刻消费丢弃（表现为玩家走不动、卡在墙外）。*

### 判定：垂直坠落机制 (Vertical Falling Check)
相较于复杂的横向碰撞，地外环境的垂直高度重力检测尤为频繁和重要：
1. 引擎每一帧在玩家足部向下发射检测垂线。
2. 在返回的碰撞堆栈中过滤所有的普通网格，**只留下所有被标记为 `stop` 的最高接触点**。
3. **如果落脚点是地块**：当射线未击中任何浮空阻挡物时，系统比较玩家底端坐标与地块底层的标高系统（`elevation`）。
4. **状态落盘**：玩家是否腾空将直接把踩踏物的唯一标识（属于哪个世界、哪个 Block 的第几个附属物）赋权给 `player.location.stop` 对象。

*(基于此机制，即便开发者用一片水 (Water) 遮住深坑，若是水没标记 `stop` 属性，玩家走过去依然会自动垂直向下坠落进入深渊。)*

## 2. 三态跨越分析表

运动状态下最易出错的即是阶梯判定（跨步），其物理演算严格遵照此跨越分析表进行事件的抛出：

| 从当前出发点 | 前方预估落脚点类型 | 引擎判定的物理动作与内部事件广播 |
|---|---|---|
| **平坦地块** (Block) | **阻拦器** (Stop) | 如果高差在 span 之内 $\rightarrow$ 登高跨越 (`on`)<br>如果过高 $\rightarrow$ 受到物理阻隔 (`stop`) |
| **阻拦器** (Stop) | 其他**阻拦器** (Stop) | 如果处于另一高度跨度内 $\rightarrow$ 攀登/下台阶 (`on`)<br>如果产生巨大高度失位 $\rightarrow$ 执行自由落体抛物 (`fall`)<br>否则受到物理阻隔 (`stop`) |
| **阻拦器** (Stop) | 回到平坦**地块** (Block) | 高度缓和 $\rightarrow$ 恢复正常降落 (`leave`)<br>如差距极大 $\rightarrow$ 产生坠落效果 (`fall`) |

## 3. 跨 Block 加载的边界碰撞容错

大世界最棘手的物理问题产生于“走到了地图边界，远处地块模型还未加载”时。
由于渲染资源未就绪，此时若使用网格作为探测体玩家将不停掉下地平线。为了规避此物理 BUG：
1. `PhysicsSystem` 工作在数据管线 (Pipeline Data) 而非最终模型。
2. 即使那片空地的草地与房子模型正在从网络下载加载 (`loading`)，只要其抽象数据包 (Raw Data) 已通过 API 被拉取，抽象的地板和空气墙拦截网格就会静待内存。
3. 当玩家强行冲刺进入未加载显示内容的 Block 时，系统将会安全地基于无形的数据框计算行走碰撞而避免重力错乱。
