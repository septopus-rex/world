# 玩家系统 (Player System)

玩家系统负责管理引擎内虚拟化身（Avatar）的状态流转、能力配置以及与环境的交互（下落、阻挡等）。

## 1. 玩家数据模型 (Player Data Structure)

Player 的核心状态机并不完全依赖于 3D 渲染表现，它的基础状态数据被严格定义为纯粹的逻辑数值组合，支持在无头环境（Headless）下进行校验：

```javascript
const player_state = {
    location: {
        block: [2025, 501],       // 玩家当前站立的所属地图块
        position: [8, 14, 0],     // 玩家在当前 Block 内的本地偏移量 (x, y, z)
        rotation: [0, 0, 0],      // 玩家视角朝向 (主视角的旋转弧度)
        world: 0,                 // 所在多重世界的 ID
        stop: {                   // 脚下正在踩踏着什么物件
            on: false,            // 悬空还是落地
            adjunct: "box",       // 若踩在物体上，这是物体的种类名称
            index: 0,             // 踩在的物体在其 Block 组内的下角标
        }
    },
    capacity: { ... },            // 玩家当下的运动能力上限
    body: { ... },                // 虚拟化身的碰撞体格描述
    posture: "stand",             // 当前动作姿态 (stand/walking/running/climbing/squatting/lying)
}
```
*注：考虑到去中心化应用的安全性，这部分核心状态数据（不涉及私钥）在浏览器环境中会被实时序列化备份到 `localStorage` 中实现断线位置续传，但不可信赖的内容后续由合约同步。*

## 2. 运动能力 (Capacity) 与身体参数 (Body)

### 运动能力 (Capacity)
不同于将速度写死在控制器中，Septopus 的玩家能够通过读取特殊 Adjunct 或触发器改变自身的能力面板。
*   `move`: 单帧直线移动距离倍率。
*   `jump`: 最大离地跳跃高度参数。
*   `span`: 可跨越障碍物的最大高度（例如 `span` 为0.3m时，即使前方有0.2m的台阶，玩家也能无需跳跃直接走上去）。
*   `death`: 坠落致死高度（当跌落差额超过此数值时，系统将发出 `player.death` 广播事件并强制复位玩家）。

### 身体参数 (Body Section)
采用分段式定义的结构树用来挂载或驱动各类 3D 头像（Avatar）。
利用 IPFS 的外部加载机制，玩家所穿戴的新模型会依据该体格骨架参数（如身高、肩宽、腿长比例）进行重定向匹配，保证碰撞框在替换模型后依然能够紧密贴合人物形态。

## 3. 状态转换机与同步

当用户的交互设备（键盘、屏幕遥感）发出诸如 “向前移动” 这样的指令时，引擎并非粗暴地直接修改对象的 3D `mesh.position`，而是严格走过以下三个周期：

1. **逻辑解算器推算位置**：`movement` 模块计算下一帧本机的理论偏移量（例如向前步进 `(x, y)`、下坠 `(z)`）。
2. **逻辑状态更迭**：系统检查这步长是否超出或者触碰了[边界检测系统](./physics.md)，将更新后的结果反馈赋值入 `env.player.location`。
3. **驱动摄像机与画面**：渲染同步机制将数据总线里最新产生的位置结果推流进当前活跃的三维摄像机（`PerspectiveCamera`）。
   - *（如果是第三人称模式或存在镜像观察摄像机，该动作还会被派发给目标模型引发诸如“跑步”之类的时间轴骨骼动画挂载）。*
