# 数据管线 (Data Pipeline) 与渲染管线

Septopus World 是一个**数据驱动**的引擎，引擎本身不直接内置任何游戏场景，所有的可视化三维内容必须经过一套标准化的多层数据管线进行解析和渲染。

## 1. 数据转换管线 (Data Transformation Pipeline)

数据的流动是从抽象的链上存储格式，一步步转换为具体的 3D 渲染对象。该流转过程分为三个核心阶段：

### 阶段一：Raw 数据 (链上存储层)
由 `datasource` 组件通过 API/RPC 从区块链或 IPFS 获取的原始序列化数据。为了极限压缩上链成本，Raw 数据通常为紧凑的数组格式。
```javascript
// Raw 数据结构的典型表现：极简数组
[
    elevation,    // 地块(Block)的基础标高
    status,       // Block 的状态位信息
    [             // Block 上挂载的所有附属物(Adjunct)数组
        [short_key, [params_data]], 
        ...
    ],           
    gameSetting   // 游戏特殊设定位
]
```

### 阶段二：STD 数据 (标准业务层格式)
Raw 数据对于渲染器和各类系统来说过于晦涩，因此会经过 `framework.structSingle()` 统一转换为标准的中间层对象（STD）。STD 数据包含了所有系统能够读懂的空间、属性和状态特征。
```javascript
// STD 数据的典型结构
{
    block: [{
        x, y, z,              // 基础尺寸
        ox, oy, oz,          // 空间相对偏移
        rx, ry, rz,          // 旋转弧度
        elevation,             // 基础标高
        material: { texture, color, repeat }, // 解析后的材质属性
        status,
        game
    }],
    wall: [...],             // 墙体类附属物集合
    trigger: [...],          // 触发器类附属物集合
    // ... 其他基于组件注册的分类
}
```

### 阶段三：3D 数据 (视图层格式)
由 `framework.structRenderData()` 将 STD 数据进一步加工。该阶段会应用坐标系转换（详见[坐标系统](./coordinate.md)），将物理逻辑系统的数据格式化为可以交由具体渲染引擎（如 Three.js 或 2D Canvas）直接装载的显示对象格式。

## 2. 渲染同步管线 (Render Pipeline)

渲染管线的核心职责是将 3D 数据映射到屏幕并处理时间流向：

### 帧同步循环 (Frame Loop)
引擎维护一个全局的渲染循环：
1. **收集活动场景**：获取当前聚焦的 DOM 容器与活动的摄像机。
2. **执行帧同步队列**：取出本帧所有注册在循环队列中的逻辑函数并顺序执行（物理计算、位置修正等）。
3. **驱动动画系统**：如果实体挂载了形如 `category: "mesh", timeline: [...]` 的动画指令，在此阶段对 Mesh 对象上的材质、位置、旋转等属性计算增量（Delta）并施加。
4. **渲染提交**：指挥活跃的渲染器进行 Draw Call 提交更新画面。

### 延迟加载与缓存管线
为了支撑超大规模的连片世界：
*   所有 Block 均基于玩家的当前位置按空间范围动态请求 Raw 数据。
*   通过流水线式的预加载（Prefetching）：`坐标进入范围 -> 加载 Raw -> 转 STD 并解析出所需依赖(纹理/模型模块) -> 加载资源 -> 转 3D -> 提交渲染`。
*   当对应目标移出视野边界，触发清理流逆向销毁 3D 对象并从内存中剔除过期数据，保证显存和内存占用的稳定。
