# ECS (Entity-Component-System) 架构设计

Septopus World 采用改进型的 ECS (实体-组件-系统) 架构模式，以实现纯数据驱动的 3D 渲染和高弹性的功能扩展。

## 核心设计概念

传统的 ECS 架构中，系统通过遍历包含特定组件的实体来执行逻辑。在 Septopus 中，这个概念被适配为适应全链运行和动态加载的需求。

### 1. 组件化架构 (Components)
*   **功能原子化**：引擎的所有功能模块均以组件化形式存在。
*   **动态注册**：组件通过挂载到 `Framework` 核心统一注册，实现插件化管理。
*   **组件分类**：
    *   `system`: 核心系统级组件（例如时间、天气）
    *   `render`: 渲染相关的组件（包含 2D 和 3D 变体）
    *   `controller`: 交互控制级组件（FPS 控制器、观察器等）
    *   `adjunct`: 在场景中实际表现存在的附属物组件（建筑、机关、NPC 的基础模块）
    *   `datasource`: 数据提供源组件（负责对接不同的链上或离线数据格式）
    *   `plugin`: 面向第三方扩展功能的插件组件

### 2. 生命周期与钩子 (Hooks)
每个注册到系统中的组件实体必须通过 `hooks` 系统来暴露自身的生命周期。引擎通过回调这些 hooks 来调度不同组件的工作：
*   **初始化阶段**：如 `reg`, `def` 用于组件自述与定义参数解析
*   **执行与响应阶段**：如 `animate`, `task` 处理帧级动画或定期任务逻辑

### 3. 事件驱动机制 (Event-Driven)
作为系统 (System) 间解耦的重要手段，Septopus 提供了一套强类型的统一事件总线：
*   **全局级系统事件**：如 `system.init`、`system.launch`、`system.update` 等，控制世界状态流转。
*   **实体级对象事件**：如 `block.loaded`、`trigger.in`、`player.fall` 等。
*   **精准事件绑定**：事件的触发可以精确指向具体的实体 Target（包含世界坐标系位置和具体的 adjunct index），避免了全局遍历的消耗。

## 面向链上数据的改造

在 Septopus 的 ECS 实现中，最大的不同在于其“实体(Entity)”数据的来源。原始的实体数据并非在引擎内创建，而是：
1. 从链上作为 Raw Array 读取出来。
2. 映射成为特定的 Component 载体（例如某个 Block 或其上的 Adjunct）。
3. 随后交由对应的 System (例如 Renderer 或 Effects System) 去解释并转换为屏幕上的表现形式。

这种高度解耦的设计确保了即便在不同的 Web3 运行环境下（不同的链、或者是纯离线测试），只要 `datasource` 提供兼容的 Raw 数据，整个核心渲染与逻辑循环就不需要任何改动。
