# SPP 多点阵形态（非盒胞形，hex / triangle prism …） — v2 设计探讨

Status: **设计探讨（未实现，长期方向，先放着）** · 2026-07-06 · 过程文档(非规范)
关联: `../../../../spp-protocol/specs/SPP-Core-v1.0.md`(协议,box-only)· `spp-protocol-full.md`(A–E 现状)·
`spp-recursive-refinement.md`(细化)· `spp-editors.md`(编辑器)。参考锚点见文内 file:line。

> **本文只做设计探讨,不排期。** 结论先行:**SPP 的核心思想形态无关**,六棱柱/三棱柱等空间铺满胞形
> **逻辑上都成立**;但**现有协议与实现是盒子专用**,换胞形 = 扩协议 + 扩引擎,且**最硬的一道是斜面碰撞**。
> 落地前须先分清"要非盒**视觉**(现在就能做)还是非盒**点阵**(这份文档的大工程)"。

## 0. 一句话

SPP = 离散节点 + 面级连接选项 + 坍缩,**这套对任何"能铺满空间、有明确共享面、有邻居坐标系"的胞形都成立**。
但协议把面写死成 6 个正交面、引擎写死成立方 + AABB 碰撞;要 hex/triangle 点阵,得系统性地扩这两层,
其中**斜面碰撞(AABB→凸体)是主成本**。

## 1. 核心洞察:概念形态无关,现状盒子专用

- **概念层(SPP 思想)**:节点 + faceOptions + collapse,不依赖胞是立方——**形态无关**。
- **协议层(SPP-Core v1.0)**:`faceStates` 是 **6-bit 掩码**(§3.2.3,±X/±Y/±Z)、`faceOptions: id[6][]`(§3.2.4)——
  **硬编码 6 正交面 = 立方点阵假设**。
- **实现层(engine)**:`ParticleFace` 6 面枚举(`core/types/ParticleCell.ts`)、`cellSize=4·0.5^level` 立方
  (`core/spp/Expander.ts`)、`FACE_DIR` 六个笛卡尔方向、`partToBox` 映到 6 盒面、碰撞 **AABB-only**
  (`Expander.ts` 头注"no cell rotation, AABB-only")。**全是盒子。**

⇒ 换胞形不是"改配置",是**动协议 + 动引擎**。

## 2. 胞形成立的三条件

任何胞形要进 SPP 模型,须同时满足:

1. **能铺满空间(space-filling tiling)**——胞与胞无缝相邻,无空隙。
2. **有明确的共享面**——每个面恰好与一个邻居共享(faceOptions/坍缩/相邻消除依赖此)。
3. **有邻居坐标系**——能索引胞、O(1) 找到各面的邻居。

立方满足(笛卡尔 + 6 面 + 正交邻居)。别的胞形只要满足这三条,SPP 的坍缩/继承/展开机制**原样适用**。

## 3. 候选点阵

| 点阵 | 胞形 | 面数 | 铺满 3D? | 邻居坐标 |
|---|---|---|---|---|
| **立方**(现状) | cube | 6 | ✅ | 笛卡尔 (x,y,z) |
| **六棱柱** | hex prism(六边形×竖直) | **8**(6 侧 + 上下) | ✅(蜂窝×层) | 六边形轴坐标 axial (q,r) + z |
| **三棱柱** | triangular prism(等边三角×竖直) | **5**(3 侧 + 上下) | ✅(三角格 up/down 交替×层) | 三角格坐标 + z |
| 菱形十二面体 | rhombic dodecahedron | 12 | ✅(FCC 填充) | FCC 索引(较复杂) |

> ⚠️ **坑:2D 三角/六边能铺满 → 对应"柱体"在 3D 能铺满;但 3D 的"正四面体"单独不能铺满空间**
> (须配正八面体的 tet-oct 蜂窝)。所以"等边三角形"当**三棱柱**没问题,当**四面体**就不行。

最实用的两个候选:**六棱柱(有机/自然/蜂窝感)** 与 **三棱柱(尖角/斜面)**。

## 4. 现状约束(逐条,决定改哪里)

| 环节 | 现状(box-only) | 锚点 |
|---|---|---|
| 面数/方向 | 6 正交面 | `ParticleFace` 枚举 · `FACE_DIR` |
| 面掩码 | 6-bit `faceStates` / `faceOptions[6]` | 协议 §3.2.3–4 |
| 胞尺寸 | `cellSize=4·0.5^level` 立方 | `Expander.ts:cellSize` |
| 面几何 | `partToBox` 映到 6 轴对齐盒面 | `Expander.ts:partToBox` |
| 相邻消除 | 笛卡尔 6 邻居 | `Expander.ts:NEGATIVE_FACE_DIR/FACE_DIR` |
| 细化 | 立方 2×2×2 子(`⌊pos/2⌋`) | `spp-recursive-refinement.md` |
| **碰撞** | **AABB-only,无胞旋转** | `Expander.ts` 头注 · `MovementCollider` |

## 5. 要支持非盒胞形,得补什么

1. **点阵描述子(lattice)**:一个字段或每 chunk 固定一种(cube / hexPrism / triPrism …)。**不入 CID 的运行时?否**——
   点阵是内容层事实,须入源与 CID(同 origin/theme,不同点阵 = 不同世界结构)。
2. **每胞形的面数 + 方向表**:六棱柱 8 面(6 侧 60° 间隔 + 上下)、三棱柱 5 面。`faceStates` 从 6-bit → **变长掩码**;
   `faceOptions` 从 `[6][]` → `[N][]`。
3. **邻居/坐标系**:六边形 axial(q,r)+z、三角格(up/down)+z;每面的邻居查询按点阵定义。
4. **面几何映射**:`partToBox` → `partToPrismFace`——把面局部 (u,v,w) 映到**倾斜的**实际面平面(60°/30° 法向)。
   part 的 unit frame 语义保留(尺度无关),只是面平面不再轴对齐。
5. **细化规则**:非立方的自相似细分(六边形有 rhombille/子六边形方案;三角有 4-子三角)——比立方 2×2×2 复杂,
   父子继承规则要重推(接 `spp-recursive-refinement.md`)。
6. ⚠️ **碰撞(见 §7)**。

## 6. 关键区分:非盒"视觉" vs 非盒"点阵"

**先分清你要哪个,否则会把小事做成大事:**

- **非盒视觉**(六角亭、三角屋顶、菱形窗、圆柱)——**现在就能做,零协议改动**:在**立方胞**里用 `parts`
  (a6 锥 / a7 球 / 带旋转的 a1·a2 / a4 外部模型)或 motif 拼出非盒外观。胞还是盒,长相随便。
- **非盒点阵**(整个世界按六边形/三角形网格**铺**)——**才是本文的大工程**:换的是点阵/铺法本身。

多数"我想要六边形的东西"其实是**前者**(视觉),用 parts/motif 即可。只有当你要**空间本身按 hex/tri 组织**
(棋盘式六边格世界、蜂窝城市底格)时,才需要**后者**(换点阵)。

## 7. 最硬的一道:斜面碰撞(AABB → OBB/凸体)

立方胞的墙面**轴对齐**,引擎的 **AABB 碰撞**(轴对齐包围盒)正好够用。非盒胞形的墙面是**斜的**(六棱柱侧面
60°、三棱柱侧面 30°/60°),AABB 无法正确表达 → 玩家会"卡"在斜墙的隐形直角上,或穿过。

要正确碰撞非盒墙,须把碰撞从 AABB 升级到 **OBB(有向包围盒)或凸多面体碰撞**——这是**比数据层大得多的引擎工程**
(`MovementCollider` 现在的 footprint/topY 都是轴对齐假设;slope stop 是特例的楔形,不是通用凸体)。**这道不解决,
非盒点阵就只是"看着对、走起来错"。** 这是本方向的**主成本与主风险**。

> 缓解路径(若先要视觉不要精确碰撞):非盒墙用**近似 AABB 碰撞体**(斜墙套一个轴对齐盒),接受"墙脚有点虚"的
> 手感,把精确凸体碰撞留到 v2.1。但这只是权宜。

## 8. 协议治理(你是 spp-protocol 作者)

- 现协议 §3.2.3 的 6-bit `faceStates` 与 §3.2.4 的 `faceOptions[6]` **写死了立方**。支持多点阵 = **协议 v2 扩展**:
  新增 `lattice` 字段 + 变长面掩码/faceOptions + 每点阵的邻居/坐标约定。
- 这是**扩协议**(不是填协议留白)——不同于 StylePack(协议主动留白、实现自主)。所以若要做,**应当先在
  `spp-protocol` 里提 `SPP-Lattice` 扩展规范**,再落引擎,免得协议与实现分叉。
- 兼容:立方是"lattice=cube"的特例;老数据无 lattice 字段 → 默认 cube,零迁移。

## 9. 分档 / 路线（若将来要做，非排期）

- **R0 只要非盒视觉** → 不做本文,用 parts/motif(已能)。
- **R1 数据模型**:`lattice` 字段 + 变长 `faces`/`faceOptions`(先支持 hexPrism 一种);展开到**近似 AABB 碰撞**
  (斜墙套轴对齐盒)——**先让它能长出来、能大致走**。
- **R2 面几何 + 邻居**:`partToPrismFace` + 六边形 axial 邻居 + 相邻消除;视觉正确。
- **R3 精确碰撞**:OBB/凸体碰撞(引擎大改),斜墙手感正确。**主成本在此。**
- **R4 细化**:非立方自相似细分 + 父子继承(接 refinement 文)。
- **协议**:R1 前在 `spp-protocol` 提 `SPP-Lattice` 扩展,双语。

## 10. 非目标 / 开放问题

- **非目标**:不规则/非周期铺(Voronoi、Penrose)——SPP 依赖规则邻居坐标,非周期铺不在本方向内。
- **开放**:六边形的**细化**方案(子六边形 vs rhombille)未定;三棱柱的 up/down 交替对坍缩 seed 的影响未推。
- **开放**:混合点阵(一个世界里 cube 区 + hex 区相接)的**界面缝合**——大概率 v3。
- **判断**:除非有明确的"空间必须按 hex/tri 组织"的玩法需求(六边格战棋、蜂窝城),否则**优先用 parts/motif 做
  非盒视觉**,把换点阵这件大事(尤其斜面碰撞)往后放。本文先放着。
